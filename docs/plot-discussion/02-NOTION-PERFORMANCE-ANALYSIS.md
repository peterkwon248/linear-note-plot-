# 노션 성능 아키텍처 분석 — "개인용 노션"은 가능한가?

> 2026-03-30 대화 기반 정리

## 핵심 질문

> 노션의 기능을 전부 구현하되, '빠르게' 작동하게 할 수 있을까?

## 노션이 느린 진짜 이유

### 전제: "모든 문단이 DB 블록"은 맞다

- 노션에서 텍스트 입력, 체크박스 토글 등 모든 행위가 블록을 생성하고, 각 블록은 PostgreSQL의 한 행으로 저장
- 키보드 ↑↓ 이동도 사실상 DB 행 사이를 점프하는 것
- 10개 요소가 있는 "단순한" 페이지 하나가 10개 이상의 DB 쿼리를 발생시킴

### 하지만 블록 구조 자체가 느린 건 아니다

느린 이유는 세 가지 레이어가 겹쳐서:

#### 1) 서버-클라이언트 왕복 (가장 큰 원인)

- 클라우드 퍼스트 → 페이지 열 때마다 서버에서 블록 트리를 받아와야 함
- `loadPageChunk`로 받은 데이터를 전부 메모리에 올리고 React로 렌더링
- 네트워크 레이턴시 + 페이로드 크기에 직격탄
- 한국에서 쓰면 미국 서버까지 왕복 → 더 느림

#### 2) 실시간 협업 동기화 오버헤드

- 편집할 때마다 operation → transaction으로 묶여서 서버로 전송, 동시 편집자들에게 브로드캐스트
- 혼자 쓸 때도 이 파이프라인이 항상 동작
- 블록 데이터의 90%가 update (insert가 아님) → 쓰기 부하 극심

#### 3) DB 구조의 스케일 문제

- DB 엔트리 하나가 단일 행이 아니라, 각 속성/관계/콘텐츠를 표현하는 여러 블록으로 구성
- 필터, 정렬, 롤업 같은 연산이 엄청난 계산량 요구
- 2021년 초 200억 블록 → 현재 2,000억 블록 이상
- 96개 DB 서버, 480개 논리 샤드로 분산해도 빠듯

## Plot이 구조적으로 피하는 것들

| 노션의 병목 | Plot의 구조 | 효과 |
|---|---|---|
| 서버 왕복 | **로컬 퍼스트 (IndexedDB)** | 네트워크 레이턴시 = 0 |
| 실시간 협업 동기화 | **싱글 유저** | 동기화 오버헤드 = 0 |
| 2,000억 블록 샤딩 | **개인 워크스페이스** | 블록 수 현실적으로 수만~수십만 |

→ 노션 느림의 **80% 이상**이 사라짐

## 주의할 구간 — "기능 자체가 느림을 내재하는" 것들

### 인라인 DB (Linked Database View) — 노션에서 가장 느린 기능

- 인라인 DB가 여러 개 있으면 동시에 데이터를 로드하고 업데이트를 리스닝
- Plot에서는 이미 거부한 기능이며, 올바른 판단

### 롤업/관계형 수식 체인

- 다른 수식에 의존하는 수식이 다시 롤업에 의존하는 체인 → 기하급수적 계산 시간
- 로컬에서 돌려도 느릴 수 있음
- 만약 수식을 넣는다면 의존성 깊이 제한(depth cap 2~3) 필수

### 블록 단위 저장 vs. 문서 단위 저장 — 핵심 설계 갈림길

- 노션: 문단 하나 = DB 행 하나
- Plot: TipTap 기반 → **문서 전체를 하나의 JSON으로 IndexedDB에 저장**
  - 페이지 로드 = IndexedDB에서 JSON 1건 읽기 (vs. 노션의 수십~수백 블록 쿼리)
  - 저장 = JSON 1건 쓰기 (vs. 노션의 블록별 트랜잭션)
  - 트레이드오프: Synced Block 같은 "다른 페이지 블록 참조" 어려움 → 이미 제외한 기능

### 위키 검색 / 백링크 인덱싱

- 노트 수 만 단위 넘으면 전문 검색 느려질 수 있음
- 별도 검색 인덱스 미리 설계 필요 → Plot은 FlexSearch Worker + IDB 인덱스로 이미 해결

## 결론

> "옵시디언이나 업노트처럼 빠르게 작동하는 개인용 노션" — 충분히 만들 수 있다.

노션이 느린 핵심 원인이 "3천만 유저 실시간 클라우드 동기화"이지, 기능 그 자체가 아니기 때문.

**빼야 할 건 빼는 게 맞다:**
- 인라인 DB 뷰 (노션 방식의 스키마 자유 정의)
- Synced Block
- 깊은 롤업 체인

이건 기능의 본질 자체가 복잡도를 내재하고 있어서, 아키텍처가 달라도 결국 문제를 일으킨다.
