# 인라인 DB — 스키마 없는 쿼리 뷰 설계

> 2026-03-30 대화 기반 정리

## 핵심 질문

> 인라인 DB가 있어도 속도를 빠르게 할 수 있는가?
> 인라인 DB가 무조건 스키마 설계를 해야 하는가?

---

## 결론: 둘 다 Yes

### 1. 로컬이면 인라인 DB는 느리지 않다

노션에서 인라인 DB가 느린 이유 4가지:
1. 인라인 DB 하나 열 때마다 **서버에 별도 쿼리** 날림
2. 한 페이지에 3개 있으면 **3번 왕복**
3. 각 DB가 **실시간 업데이트를 리스닝** (웹소켓 연결 유지)
4. 다른 유저의 수정사항을 **브로드캐스트 수신**

→ 전부 네트워크 + 멀티유저 문제. 로컬이면 전부 사라짐.

로컬에서 인라인 DB가 하는 일:
```
IndexedDB에서 메타데이터 쿼리 → 필터/정렬 → React로 테이블 렌더
```
→ **수 밀리초** 안에 끝남. 노트 1만 개여도.

### 2. 스키마 설계 없이 인라인 DB를 만들 수 있다

노션 방식: "빈 테이블 만들고 → 컬럼 추가 → 타입 지정 → 데이터 입력"
Plot 방식: **시스템이 미리 정의한 스키마를 활용**

Plot의 모든 노트에 이미 있는 메타데이터:
- NoteStatus (inbox / capture / permanent)
- Label (1:1, 노트 타입)
- Tags (N:N)
- 생성일 / 수정일
- 폴더 경로
- 관계(Relations) — 위키 크로스링크, 백링크

---

## 설계: 스키마 없는 인라인 쿼리 뷰

### 사용 흐름

```
에디터에서 /query 치면 →
"이 폴더의 노트들" or "이 태그의 노트들" or "이 위키에 연결된 노트들"
→ 테이블/칸반/리스트로 바로 렌더
```

유저가 하는 건 **"뭘 보여줄지" 선택하는 것뿐**. 스키마를 만드는 게 아님. 필터 조건만 고르면 됨.

### 기존 인프라 재활용

- 뷰 엔진 `runPipeline()` (context→filter→search→sort→group) 이미 존재
- TipTap 커스텀 노드로 감싸기만 하면 됨

### 우려 해소

| 우려 | 해소 |
|------|------|
| 구현 복잡도 | 스키마 엔진/수식/롤업 불필요. 이미 있는 메타데이터 쿼리만 |
| 철학 충돌 | 유저가 DB 설계 안 함. 그냥 쓴 노트를 다르게 "보는" 것 |
| 성능 | 메타데이터 필터링이니까 밀리초 단위 |

### "무의식적 제텔카스텐"과의 정합성

유저는 그냥 노트를 쓰고, 어느 순간 에디터 안에서 `/query` 해보면 자기가 쌓아온 지식이 테이블로 펼쳐지는 경험. 스키마 설계의 부담 제로.

## 세 가지 선택지 비교

| 선택지 | 설명 | 구현 난이도 | 철학 적합성 |
|--------|------|------------|------------|
| A) 안 한다 | 뷰는 Activity Bar 레벨에서만 | - | ○ |
| **B) 스키마 없는 쿼리 뷰** | 에디터 안에서 기존 메타데이터 쿼리 + 렌더 | 중간 | **◎** |
| C) 풀 인라인 DB (노션 카피) | 유저 정의 스키마 + 수식 + 롤업 | 3~6개월 | △ |

→ **B 채택**: 인라인 DB ≠ 스키마 설계 필수라는 전제를 깔면, Plot에 넣을 수 있고, 빠르고, 철학도 안 깨짐.
